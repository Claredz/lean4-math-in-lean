```lean
example (x y z : ℝ) (h₀ : x ≤ y) (h₁ : y ≤ z) : x ≤ z := by

  apply le_trans h₀

  apply h₁
  
example (x y z : ℝ) (h₀ : x ≤ y) (h₁ : y ≤ z) : x ≤ z :=

  le_trans h₀ h₁
```
这两种使用定理的方式在侧边栏里面的内容不太一样

我们先从 apply 一个定理后会有什么变化开始，

以我目前的理解，apply 一个定理之后 lean 会去尝试**匹配**符合定理形式的现有的内容

在第一个例子里面,
在 apply le_trans 之后出现了 3 个 goals

> [!example]
> 3 goals
> case a
> a b c d e x y z : ℝ
> h₀ : x ≤ y
> h₁ : y ≤ z
> ⊢ x ≤ ?b
> case a
> a b c d e x y z : ℝ
> h₀ : x ≤ y
> h₁ : y ≤ z
> ⊢ ?b ≤ z
> case b
> a b c d e x y z : ℝ
> h₀ : x ≤ y
> h₁ : y ≤ z
> ⊢ ℝ

事实上，这是匹配的结果，原始定理是这样的

le_trans.{u_1} {α : Type u_1} [Preorder α] {a b c : α} : a ≤ b → b ≤ c → a ≤ c

事实上，在第一个例子里面只有 x ≤ z 与a ≤ c 匹配于是 x=a, z=c, 但是没有一个合适的量匹配 b, 于是在 goals 里面出现了？b, 在我们提供更多内容后 ？b=y 了

但是第三个 caseb 在干什么？它是在确定什么的类型为 R？

GPT 的解释是这样的
	第三个 `⊢ ℝ`，本质上不是又多出来一个数学条件，而是 **Lean 还在处理 `le_trans` 里的隐含参数**，尤其是它作用在哪个类型 α\alphaα 上。  
	虽然从上下文看应该是 R\mathbb RR，但在 `apply` 的这个瞬间，它可能先把这个类型参数也当成一个待确定的 hole，所以面板里临时显示成了第三个目标。
	
	所以你看到的 3 个目标，实质上是：
	
	- 两个真正的序关系证明目标；
	- 一个隐参数/类型参数的暂时未定项。

貌似如果深究，必须要搞清楚在 lean 4 里面**定理**究竟是什么，它的存在形式是什么？是函数？类型？

之前在侧边栏里面，出现了类似类型的特征

⊢ ℝ和⊢ ?b ≤ z 属于是同种结构，也与我查到的**命题即类型符合**
